🔪 今日锐评

推荐系统终于迎来了自己的 scaling law 统一架构（UniMixer），而 Agent 赛道继续高歌猛进——从训练框架到对话系统到多模态生成全线开花。今天的亮点是工业界和学术界同时发力：快手的 UniMixer 把三条推荐 scaling 路线统一了，微软的 Agent Lightning 拿下 140 upvotes 靠的是真正解耦了 agent 执行和 RL 训练。质量两极分化依旧：头部论文有扎实的工程和实验，尾部的金融 multi-agent 们还在用 GPT 套壳讲故事。

1. UniMixer: A Unified Architecture for Scaling Laws in Recommendation Systems

  • 作者: Mingming Ha, Guanchen Wang, Linxun Chen, Xuan Rao, Yuexin Shi, Tianbao Ma 等
  • 机构: 快手
  • 链接: arXiv | PDF
  • 来源: 📰 HF Daily，⬆️ 3
  • 核心方法: 将推荐系统 scaling 的三大范式——attention-based、TokenMixer-based、FM-based——统一到 UniMixing 框架下。核心观察：三种方法本质上都是对特征交互矩阵做不同约束的混合操作。提出 RankMixer 作为统一实现，引入 Sinkhorn-Knopp 双随机约束和 SiameseNorm 控制参数膨胀。在工业级数据上验证了统一 scaling law。
  • 对比方法/Baselines: 对比: HiFormer, attention-based CTR, FM-based methods；评测: AUC, UAUC, DeltaAUC
  • 借鉴意义: 推荐系统终于有了像 NLP 的 Chinchilla 一样的统一 scaling 分析框架。对理解"推荐模型该怎么扩"有直接指导价值。RankMixer 的设计思路可直接用于 CTR 预估模型的架构选型。
  • 锐评: 快手出品，工业验证是最大加分项。把三条看似不同的技术路线统一到一个框架下，理论推导有说服力。但 3 个 upvotes 说明大多数人没看懂这篇的价值——这是今天被低估最严重的论文。缺点是 RankMixer 的计算开销对比分析不够清晰，Sinkhorn-Knopp 在大规模稀疏场景下的收敛性讨论缺席。 🔥

    • 2. MemRerank: Preference Memory for Personalized Product Reranking

  • 作者: Zhiyuan Peng
  • 机构: Santa Clara University
  • 链接: arXiv | PDF
  • 来源: 📰 HF Daily，⬆️ 2
  • 核心方法: 将用户购买历史蒸馏为 query-independent 的偏好记忆（preference memory），用于 LLM-based 商品重排序。MemAgent 负责记忆提取，GRPO 做强化学习优化。核心思路：把噪声、冗长的购买记录压缩为简洁的偏好信号，避免直接拼接历史到 prompt。
  • 对比方法/Baselines: 对比: GPT-4, GPT-5；评测: Amazon-Review
  • 借鉴意义: "记忆蒸馏"的思路对任何需要长期用户建模的推荐/个性化系统都有参考价值。GRPO 用于 agent 记忆优化是个有趣的交叉点。
  • 锐评: 想法方向对——用记忆代替暴力拼接历史。但只在 Amazon-Review 上做实验，baseline 居然包含 GPT-5，这评测设计让人困惑。单作者论文，实验规模和深度都需要加强。 👀

    3. Agent Lightning: Train ANY AI Agents with Reinforcement Learning

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang 等
  • 机构: Microsoft Research
  • 链接: arXiv | PDF
  • 来源: 🔥 HF Trending，⬆️ 140
  • 📒 笔记: Agent Lightning - Train AI Agents with RL
  • 核心方法: 完全解耦 agent 执行与 RL 训练。LightningRL 框架将 agent 运行时抽象为 MDP，通过 OpenTelemetry 收集 trajectory，异步送入训练管线。支持任意 agent 框架（LangChain, AutoGen, MCP）无需修改代码即可接入 RL 训练。
  • 对比方法/Baselines: 对比: GRPO, PPO；评测: MuSiQue, SQL, QA benchmarks
  • 借鉴意义: "执行-训练解耦"是把 RL 训练从实验室搬到生产环境的关键一步。OpenTelemetry 做 trajectory 采集的思路非常工程化，值得直接借鉴。
  • 锐评: 140 upvotes，微软出品。和 OpenClaw-RL 思路相似但工程化程度更高——用 OpenTelemetry 这种工业级可观测性工具收集训练信号，这才是做框架该有的样子。但"Train ANY AI Agents"这个标题和 OpenClaw 一样在吹牛，对 long-horizon 任务的 reward 稀疏问题完全没讨论。 🔥

    6. GEMS: Agent-Native Multimodal Generation with Memory and Skills

  • 作者: Zefeng He, Siyuan Huang, Xiaoye Qu, Yafu Li, Tong Zhu, Yu Cheng, Yang Yang
  • 机构: Shanghai AI Laboratory, Nanjing University, SJTU, CUHK
  • 链接: arXiv | PDF
  • 来源: 🔥 HF Trending，⬆️ 74
  • 📒 笔记: GEMS - Agent-Native Multimodal Generation with Memory and Skills
  • 核心方法: 三层 agent 架构做多模态生成的 inference-time scaling。Agent Loop（Planner/Decomposer/Verifier/Refiner 闭环）、Agent Memory（层次化 trajectory-level 记忆）、Agent Skill（可扩展技能集合）。6B 模型套 agent 框架超越 SOTA 大模型。
  • 对比方法/Baselines: 对比: Qwen-Image, GPT；评测: GenEval2, SpatialGenEval, DPG
  • 借鉴意义: Memory 分层设计和 Skill 按需加载机制可直接借鉴。证明小模型+agent框架>大模型单推。
  • 锐评: "Agent-Native"这个词造得精准。6B 干翻 SOTA 确实漂亮。但四个 agent 每轮迭代都调 MLLM，推理成本一个字没提。Memory 压缩细节和 scalability 需要看全文确认。 🔥

  • 对比方法/Baselines: 摘要未详述
  • 借鉴意义: "生成前先搜索"的范式对任何需要外部知识的生成任务（广告创意、商品图）都有直接参考价值。

    8. Multi-Agent Collaboration via Evolving Orchestration

  • 作者: Yufan Dang 等
  • 机构: Tsinghua University, SJTU, Siemens, Tencent Robotics X
  • 链接: arXiv | PDF
  • 来源: 🔥 HF Trending，⬆️ 8

  • 借鉴意义: 用 RL 优化 multi-agent 编排的思路比手工设计拓扑靠谱得多。中心化 vs 去中心化编排的 trade-off 讨论有参考价值。
  • 锐评: 清华+西门子+腾讯的阵容。"进化式编排"方向对，但 8 个 upvotes 说明实验结果可能没那么惊艳。在 MMLU 这种"一个 agent 就能做"的任务上评估 multi-agent 系统，benchmark 选择存疑。 👀

    9. A Survey of On-Policy Distillation for Large Language Models

  • 作者: Mingyang Song, Mao Zheng
  • 机构: Tencent
  • 链接: arXiv | PDF
  • 来源: 📰 HF Daily，⬆️ 4
  • 核心方法: 系统梳理 LLM 的 on-policy distillation 方法。从 exposure bias 出发，用 f-divergence 统一框架分析 GKD, DistiLLM, MiniLLM, DKL 等方法。区分 sequence-level 和 token-level 蒸馏。
  • 对比方法/Baselines: 综述覆盖: SFT, DPO, GRPO, REINFORCE
  • 借鉴意义: 做模型蒸馏的团队可以当 reference sheet 用。f-divergence 统一视角对理解各方法的本质区别有帮助。
  • 锐评: 腾讯出品的整理工作，该有的都有。但 survey 在 2026 年不稀缺，关键看能不能提供超出简单分类的 insight。从摘要看，贡献主要在整理而非发现。 💤

    • 10. Embarrassingly Simple Self-Distillation Improves Code Generation

  • 作者: Ruixiang Zhang 等
  • 机构: Apple
  • 链接: arXiv | PDF
  • 来源: 📰 HF Daily，⬆️ 11
  • 核心方法: SSD（Simple Self-Distillation）：从模型自身采样代码解决方案，不做任何验证，直接用标准 SFT 微调。无需 verifier、teacher model 或 RL。在 LiveCodeBench, HumanEval, CruxEval 上提升 Qwen3-32B 的代码能力。
  • 对比方法/Baselines: 对比: RL-based methods, DeepSeek；评测: LiveCodeBench, HumanEval, AIME
  • 借鉴意义: "不筛选、不验证、直接蒸馏"的极简方案，如果真的 work，对缺少 reward model 的团队是福音。
  • 锐评: Apple 出品，标题里的"Embarrassingly Simple"是实话。方法确实简单到令人不安——采样+SFT，连 filtering 都不做。但正因如此，claim 需要更严格的验证：为什么"不筛"比"筛了再训"好？这个反直觉的结论需要更深入的 ablation。 💤
