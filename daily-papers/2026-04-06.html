    SKILL0: In-Context Agentic Reinforcement Learning for Skill Internalization

    一句话总结：提出 "Skills at training, zero at inference" 范式——用渐进式课程学习 RL 将 agent 技能从推理时 context 注入内化到模型参数，实现零检索的自主行为。

    1. 研究动机与问题定义

    推理时技能增强的三大局限

    1. 检索噪声：不相关技能污染 agent context

    2. Token 开销：技能内容在多轮交互中累积，限制可扩展性

    3. 根本问题：模型在"执行"技能而非"学会"技能——能力在 context 中，不在参数里

    类比人类学习

    人类技能习得遵循：显式指导阶段 → 内化阶段（Anderson, 1982）。推理时技能增强永远锚定在第一阶段。RL 提供了通往第二阶段的路径。

    2. 方法详解

    2.1 Agent Loop 形式化

    任务建模为序列决策：给定指令 $I$，agent 生成动作序列 $\{a_1, a_2, \ldots, a_T\}$。每步 $t$，agent 从策略采样：

    $$a_t \sim \pi_\theta(a_t | I, h_t), \quad h_t = \{o_1, o_2, \ldots, o_t\}$$

    环境转移：$o_{t+1} = \mathcal{E}(o_t, a_t)$

    2.2 SkillBank：层次化技能库

    技能组织为两层结构：

  • 通用技能：跨任务的策略原则（探索策略、目标跟踪等）
  • 任务特定技能：领域专用知识（动作序列、前置条件等）

    • 目录结构：skills/{task_name}/{skill_category}.md

    2.3 Relevance-Driven Skill Grouping

    离线按类别分组技能，与交互历史一起渲染为紧凑的视觉 context，教模型工具调用和多轮任务完成。

    2.4 In-Context Reinforcement Learning (ICRL)

    核心创新——技能在训练时作为 in-context 指导提供，在推理时完全移除：

    $$\mathcal{L}_{\text{ICRL}} = \mathbb{E}_{\tau \sim \pi_\theta} \left[ \sum_{t} \min\left( r_t(\theta) \hat{A}_t, \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon) \hat{A}_t \right) \right]$$

    训练 rollout 中包含技能 context $\mathcal{S}_k$，但推理时 $\mathcal{S}_k = \emptyset$。

    2.5 Dynamic Curriculum（动态课程）

    不是固定退火计划，而是自适应内化：

    对每个技能文件 $\mathcal{S}_k$，评估其on-policy 有用性——比较有无该技能时 agent 在匹配验证子任务上的表现：

    $$\text{Helpfulness}(\mathcal{S}_k) = \text{Perf}(\pi_\theta | \mathcal{S}_k) - \text{Perf}(\pi_\theta | \emptyset)$$
  • 如果 $\text{Helpfulness}(\mathcal{S}_k) > \delta$：保留该技能（模型仍需它）
  • 如果 $\text{Helpfulness}(\mathcal{S}_k) \leq \delta$：移除该技能（已内化）

    • 预算线性衰减直至归零，agent 最终完全零技能运行。

    2.6 与标准 RL 的区别

    $$\text{Standard RL}: \quad \pi_\theta(a_t | I, h_t)$$
    $$\text{Skill0 训练}: \quad \pi_\theta(a_t | I, h_t, \mathcal{S}_k)$$
    $$\text{Skill0 推理}: \quad \pi_\theta(a_t | I, h_t)$$

    优点

    Agent Lightning: Train ANY AI Agents with Reinforcement Learning

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang
  • 机构: Microsoft Research
  • 时间: 2025年8月

  • arXiv: 2508.03680
  • GitHub: microsoft/agent-lightning (16k+ stars)
  • 项目主页: Microsoft Research

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：
    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$
  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：
    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：
    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{<j}) \cdot A_j \right]$$

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    2. Lightning Client: 封装 Agent 运行时（agent runtime），包含两个功能模块：

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    • 关键优势：

  • Agent-agnostic 训练: Server 只关心 LLM 优化和硬件资源管理
  • Trainer-agnostic 执行: Client 端 Agent 不依赖训练框架的实现细节
  • 利用 Observability 框架: 将 OpenTelemetry 等可观测性框架引入训练流程，实现标准化的 trajectory 收集
  • 零代码修改: 支持 LangChain、OpenAI Agents SDK、AutoGen 等主流框架开发的 Agent

    核心贡献

    1. 首个完全解耦的 Agent RL 训练框架: Agent Lightning 是第一个实现 Agent 执行与 RL 训练完全分离的框架，支持任意 AI Agent 的 RL 训练，几乎零代码修改。

    2. 统一数据接口与 MDP 建模: 通过将 Agent 执行建模为 POMDP，定义了通用的数据接口，抽象了不同 Agent 的执行逻辑差异。

    3. LightningRL 层次化算法: 提出了基于 transition 分解的层次化 RL 方法，配合 Credit Assignment Module，支持多 Agent、动态工作流等复杂交互逻辑。

    4. Training-Agent Disaggregation 架构: 通过 Server/Client 分离设计，实现了训练框架与 Agent 框架的双向无关性，并引入可观测性框架提升可扩展性。

    5. 跨框架验证: 在 LangChain、OpenAI Agents SDK、AutoGen 三个主流框架上验证了通用性。

    局限性与思考

    局限性

  • 实验规模有限: 仅在 3B 模型上验证，更大规模模型的效果尚未探索
  • 任务类型覆盖: 虽然涵盖了 text-to-SQL、RAG、math tool-use，但未涉及 GUI 交互、长期规划等更复杂的 Agent 场景
  • Credit Assignment: 论文提出了 credit assignment module，但具体的分配策略如何影响训练效果缺乏深入分析
  • 与 concatenation 方法的定量对比: Discussion 中定性分析了 transition-based 相比 concatenation-based 的优势，但缺少直接的定量 ablation

    • 个人思考

  • Agent Lightning 的解耦设计是其最大亮点。将 Agent 执行视为黑盒、仅通过标准化接口收集数据的思路，极大降低了 RL 训练的工程门槛
  • Observability 框架（OpenTelemetry）的引入很有前瞻性，将生产环境的监控基础设施复用到训练数据收集中
  • 与 RAGEN、Search-R1、Trinity-RFT 等 concatenation-based 方法的对比是后续工作的重要方向
  • 在更复杂的 Agent 场景（如 multi-agent debate、code generation pipeline）中的效果值得期待

    GrandCode: Achieving Grandmaster Level in Competitive Programming via Agentic RL

    一句话总结：首个在 Codeforces 实时比赛中击败所有人类选手（包括传奇 Grandmaster）的 AI 系统，通过多 agent RL + Agentic GRPO 实现。

    1. 研究动机与问题定义

    竞赛编程的 AI 进展

  • AlphaCode: ~1300 rating (top 54%)
  • AlphaCode2: top 15%
  • OpenAI o3: 全球第 175 名
  • Gemini 3 Deep Think: 第 8 名（但在历史题目上评测，非实时比赛）

    • 关键差距：此前没有 AI 系统在实时比赛条件下持续击败顶级人类选手。

    2. 方法详解

    2.1 Multi-Agent 架构

    GrandCode 由多个协作 agent 组成：

  • Hypothesis Model：提出结构性猜想
  • Main Solver：核心推理和解法生成
  • Summarization Model：维护长 context 的紧凑记忆
  • Test Case Generator：生成边界测试用例挑战方案

    • 形成 推理 → 验证 → 反馈 的 agentic loop。

    2.2 Agentic GRPO

    针对多阶段 agent rollout 的 GRPO 变体，解决两个问题：

    延迟奖励：竞赛编程中只有最终提交才有反馈，中间步骤无奖励。

    Off-policy 漂移：多轮 agent 交互中，早期步骤的策略与当前策略差异大。

    Agentic GRPO 结合即时奖励更新和延迟校正：

    $$\mathcal{L}_{\text{Agentic-GRPO}} = \mathcal{L}_{\text{immediate}} + \lambda \cdot \mathcal{L}_{\text{delayed\_correction}}$$

    即时奖励部分在每个 agent 阶段提供信号，延迟校正在最终评估后回溯调整。

    2.3 Post-Training + Test-Time RL

    Post-Training：在大量竞赛编程数据上进行 RL 训练

    Test-Time RL：在比赛中实时 RL 优化——对候选方案进行 best-of-N 采样，用 code reward 验证：

    $$R(\text{solution}) = \begin{cases} 1 & \text{if all test cases pass} \\ 0 & \text{otherwise} \end{cases}$$

    通过 smoothing best-of-N reward 提升训练稳定性。

    2.4 实时比赛策略

    平衡 直接生成 和 test-time RL：

  • 简单题：直接生成，快速提交
  • 难题：启动 agentic loop + test-time RL

    SODA: Semi On-Policy Black-Box Distillation for LLMs

    一句话总结：在黑盒 LLM 蒸馏中提出 "Semi On-Policy" 折中——混合使用 on-policy 和 off-policy 数据，平衡训练稳定性和学生模型纠错能力。

    1. 研究动机

    黑盒蒸馏的两难：

  • Off-policy（如 SeqKD）：从教师采样，简单稳定，但无法纠正学生固有错误
  • On-policy（如 GAD）：从学生采样+对抗训练，能纠错但不稳定且计算量大

    • 2. 方法详解

    Semi On-Policy 策略

    混合采样：以概率 $\alpha$ 从教师采样，概率 $1-\alpha$ 从学生采样：

    $$x_t \sim \begin{cases} p_{\text{teacher}}(\cdot | x_{<t}) & \text{w.p. } \alpha \\ p_{\text{student}}(\cdot | x_{<t}) & \text{w.p. } 1-\alpha \end{cases}$$

    蒸馏损失：

    $$\mathcal{L}_{\text{SODA}} = -\mathbb{E}_{x \sim p_{\text{mix}}} \left[ \sum_t \log p_{\text{student}}(x_t | x_{<t}) \right]$$

    $\alpha$ 可以是固定值或按训练进度退火（先 off-policy 后逐渐 on-policy）。

    与 DPO 的区别

    SODA 是生成式蒸馏（学生生成 → 教师评分），DPO 是偏好对齐（正负样本对比）。两者互补。

    3. 核心概念

  • Knowledge Distillation、SeqKD、GAD、DPO、On-Policy Learning

    • 4. 优缺点

    优点: 思路自然，工程实用，计算成本可控

    缺点: 创新性有限（mixed policy 不算新），需看实验细节判断贡献大小

  • 核心方法: 黑盒知识蒸馏的 off-policy vs on-policy 困境——SeqKD 等 off-policy 方法无法纠正学生的固有错误，GAD 等 on-policy 方法训练不稳定且计算量大。SODA 提出 "Semi On-Policy" 折中：部分使用 on-policy 数据，部分使用 off-policy 数据，在稳定性和纠错能力之间取得平衡。
  • 对比方法/Baselines: SeqKD, GAD, DPO
  • 借鉴意义: "Semi On-Policy" 的折中思路在 agent 训练中也适用——完全 on-policy 太贵，完全 off-policy 效果差。
  • 锐评: 实际上是蒸馏领域一个很自然的想法——mixed policy。关键看实验是否扎实、和 DPO 系列方法的对比是否公平。多机构合作但论文日期是 2026-04-04，说明刚出来。 👀

    SKILL0: In-Context Agentic Reinforcement Learning for Skill Internalization

    一句话总结：提出 "Skills at training, zero at inference" 范式——用渐进式课程学习 RL 将 agent 技能从推理时 context 注入内化到模型参数，实现零检索的自主行为。

    1. 研究动机与问题定义

    推理时技能增强的三大局限

    1. 检索噪声：不相关技能污染 agent context

    2. Token 开销：技能内容在多轮交互中累积，限制可扩展性

    3. 根本问题：模型在"执行"技能而非"学会"技能——能力在 context 中，不在参数里

    类比人类学习

    人类技能习得遵循：显式指导阶段 → 内化阶段（Anderson, 1982）。推理时技能增强永远锚定在第一阶段。RL 提供了通往第二阶段的路径。

    2. 方法详解

    2.1 Agent Loop 形式化

    任务建模为序列决策：给定指令 $I$，agent 生成动作序列 $\{a_1, a_2, \ldots, a_T\}$。每步 $t$，agent 从策略采样：

    $$a_t \sim \pi_\theta(a_t | I, h_t), \quad h_t = \{o_1, o_2, \ldots, o_t\}$$

    环境转移：$o_{t+1} = \mathcal{E}(o_t, a_t)$

    2.2 SkillBank：层次化技能库

    技能组织为两层结构：

  • 通用技能：跨任务的策略原则（探索策略、目标跟踪等）
  • 任务特定技能：领域专用知识（动作序列、前置条件等）

    • 目录结构：skills/{task_name}/{skill_category}.md

    2.3 Relevance-Driven Skill Grouping

    离线按类别分组技能，与交互历史一起渲染为紧凑的视觉 context，教模型工具调用和多轮任务完成。

    2.4 In-Context Reinforcement Learning (ICRL)

    核心创新——技能在训练时作为 in-context 指导提供，在推理时完全移除：

    $$\mathcal{L}_{\text{ICRL}} = \mathbb{E}_{\tau \sim \pi_\theta} \left[ \sum_{t} \min\left( r_t(\theta) \hat{A}_t, \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon) \hat{A}_t \right) \right]$$

    训练 rollout 中包含技能 context $\mathcal{S}_k$，但推理时 $\mathcal{S}_k = \emptyset$。

    2.5 Dynamic Curriculum（动态课程）

    不是固定退火计划，而是自适应内化：

    对每个技能文件 $\mathcal{S}_k$，评估其on-policy 有用性——比较有无该技能时 agent 在匹配验证子任务上的表现：

    $$\text{Helpfulness}(\mathcal{S}_k) = \text{Perf}(\pi_\theta | \mathcal{S}_k) - \text{Perf}(\pi_\theta | \emptyset)$$
  • 如果 $\text{Helpfulness}(\mathcal{S}_k) > \delta$：保留该技能（模型仍需它）
  • 如果 $\text{Helpfulness}(\mathcal{S}_k) \leq \delta$：移除该技能（已内化）

    • 预算线性衰减直至归零，agent 最终完全零技能运行。

    2.6 与标准 RL 的区别

    $$\text{Standard RL}: \quad \pi_\theta(a_t | I, h_t)$$
    $$\text{Skill0 训练}: \quad \pi_\theta(a_t | I, h_t, \mathcal{S}_k)$$
    $$\text{Skill0 推理}: \quad \pi_\theta(a_t | I, h_t)$$

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：
    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$
  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：
    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：
    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{<j}) \cdot A_j \right]$$

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    2. Lightning Client: 封装 Agent 运行时（agent runtime），包含两个功能模块：

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    • 关键优势：

  • Agent-agnostic 训练: Server 只关心 LLM 优化和硬件资源管理
  • Trainer-agnostic 执行: Client 端 Agent 不依赖训练框架的实现细节
  • 利用 Observability 框架: 将 OpenTelemetry 等可观测性框架引入训练流程，实现标准化的 trajectory 收集
  • 零代码修改: 支持 LangChain、OpenAI Agents SDK、AutoGen 等主流框架开发的 Agent

    一句话总结：首个在 Codeforces 实时比赛中击败所有人类选手（包括传奇 Grandmaster）的 AI 系统，通过多 agent RL + Agentic GRPO 实现。

  • AlphaCode2: top 15%
  • OpenAI o3: 全球第 175 名

    • 关键差距：此前没有 AI 系统在实时比赛条件下持续击败顶级人类选手。

    2. 方法详解

    2.1 Multi-Agent 架构

    GrandCode 由多个协作 agent 组成：

  • Hypothesis Model：提出结构性猜想
  • Main Solver：核心推理和解法生成
  • Summarization Model：维护长 context 的紧凑记忆
  • Test Case Generator：生成边界测试用例挑战方案

    • 形成 推理 → 验证 → 反馈 的 agentic loop。

    2.2 Agentic GRPO

    针对多阶段 agent rollout 的 GRPO 变体，解决两个问题：

    延迟奖励：竞赛编程中只有最终提交才有反馈，中间步骤无奖励。

    Off-policy 漂移：多轮 agent 交互中，早期步骤的策略与当前策略差异大。

    Agentic GRPO 结合即时奖励更新和延迟校正：

    $$\mathcal{L}_{\text{Agentic-GRPO}} = \mathcal{L}_{\text{immediate}} + \lambda \cdot \mathcal{L}_{\text{delayed\_correction}}$$

    即时奖励部分在每个 agent 阶段提供信号，延迟校正在最终评估后回溯调整。

    2.3 Post-Training + Test-Time RL

    Post-Training：在大量竞赛编程数据上进行 RL 训练

    Test-Time RL：在比赛中实时 RL 优化——对候选方案进行 best-of-N 采样，用 code reward 验证：

    $$R(\text{solution}) = \begin{cases} 1 & \text{if all test cases pass} \\ 0 & \text{otherwise} \end{cases}$$

    通过 smoothing best-of-N reward 提升训练稳定性。

    2.4 实时比赛策略

    平衡 直接生成 和 test-time RL：

  • 简单题：直接生成，快速提交
  • 难题：启动 agentic loop + test-time RL

    SODA: Semi On-Policy Black-Box Distillation for LLMs

    一句话总结：在黑盒 LLM 蒸馏中提出 "Semi On-Policy" 折中——混合使用 on-policy 和 off-policy 数据，平衡训练稳定性和学生模型纠错能力。

    1. 研究动机

    黑盒蒸馏的两难：

  • Off-policy（如 SeqKD）：从教师采样，简单稳定，但无法纠正学生固有错误
  • On-policy（如 GAD）：从学生采样+对抗训练，能纠错但不稳定且计算量大

    • 2. 方法详解

    Semi On-Policy 策略

    混合采样：以概率 $\alpha$ 从教师采样，概率 $1-\alpha$ 从学生采样：

    $$x_t \sim \begin{cases} p_{\text{teacher}}(\cdot | x_{<t}) & \text{w.p. } \alpha \\ p_{\text{student}}(\cdot | x_{<t}) & \text{w.p. } 1-\alpha \end{cases}$$

    蒸馏损失：

    $$\mathcal{L}_{\text{SODA}} = -\mathbb{E}_{x \sim p_{\text{mix}}} \left[ \sum_t \log p_{\text{student}}(x_t | x_{<t}) \right]$$

    $\alpha$ 可以是固定值或按训练进度退火（先 off-policy 后逐渐 on-policy）。

    与 DPO 的区别

    SODA 是生成式蒸馏（学生生成 → 教师评分），DPO 是偏好对齐（正负样本对比）。两者互补。

    3. 核心概念

  • Knowledge Distillation、SeqKD、GAD、DPO、On-Policy Learning

    • 4. 优缺点

    优点: 思路自然，工程实用，计算成本可控

    缺点: 创新性有限（mixed policy 不算新），需看实验细节判断贡献大小

  • 核心方法: 黑盒知识蒸馏的 off-policy vs on-policy 困境——SeqKD 等 off-policy 方法无法纠正学生的固有错误，GAD 等 on-policy 方法训练不稳定且计算量大。SODA 提出 "Semi On-Policy" 折中：部分使用 on-policy 数据，部分使用 off-policy 数据，在稳定性和纠错能力之间取得平衡。
  • 对比方法/Baselines: SeqKD, GAD, DPO
  • 借鉴意义: "Semi On-Policy" 的折中思路在 agent 训练中也适用——完全 on-policy 太贵，完全 off-policy 效果差。
  • 锐评: 实际上是蒸馏领域一个很自然的想法——mixed policy。关键看实验是否扎实、和 DPO 系列方法的对比是否公平。多机构合作但论文日期是 2026-04-04，说明刚出来。 👀

    现有 multi-agent 框架（如 LangChain, AutoGen）在以下方面存在不足：

    2. 可观测性差：大规模部署时难以监控和调试

    3. 长期记忆缺失：agent 无法保持跨会话的上下文

    🔧 方法详解

    核心架构

    AgentScope 1.0 的核心抽象：

    1. MsgHub（消息总线）：agent 间消息传递的统一接口，支持广播和点对点通信

    2. StateModule（状态管理）：统一管理 agent 状态，支持序列化和恢复

    3. Non-Invasive Monitoring：基于 OpenTelemetry 的无侵入式监控

    MCP 原生支持

  • Agent 可通过 MCP 协议访问外部工具和数据源
  • 工具集成不需要修改 agent 代码（non-invasive）
  • 支持 MCP server 的自动发现和注册

  • LongTermMemoryBase：长期记忆，支持跨会话信息持久化
  • 记忆检索支持语义搜索和结构化查询

  • 支持自定义 tool 注册
  • 内置错误处理和重试机制

  • 基于 gRPC 的分布式通信
  • 支持 agent 在多节点间调度
  • MsgHub 支持分布式消息广播
