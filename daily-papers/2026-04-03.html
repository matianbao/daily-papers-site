    CoALM: A Unified Conversational Agentic Language Model

    📋 基本信息

  • 作者: Emre Can Acikgoz, Jeremiah Greer, Akul Datta, Ze Yang, William Zeng, Oussama Elachqar, Emmanouil Koukoumidis, Dilek Hakkani-Tür, Gokhan Tur
  • 机构: University of Illinois Urbana-Champaign, Oumi
  • 链接: arXiv | PDF
  • 项目主页: https://emrecanacikgoz.github.io/CoALM/

    • 🎯 研究动机与问题

    当前 LLM agent 领域存在一个核心矛盾：

    1. Task-Oriented Dialogue (TOD) 系统：擅长多轮对话管理和状态追踪，但只能处理有限的 API 集合（如 book_hotel, query_restaurant），面对新服务需要昂贵的重训练

    2. Language Agents (LA)：擅长 function calling，能处理多样化的 API，但无法维持多轮对话中的用户意图

    本文的核心问题是：能否用一个统一模型同时掌握多轮对话和工具调用？

    作者在三个 benchmark 上验证了这一矛盾的存在：

  • MultiWOZ 2.4（TOD benchmark）：LA 模型如 ToolAce 的 Success Rate 仅 18.0%
  • BFCL V3（LA benchmark）：TOD 模型如 FNCTOD 的 Overall Acc 极低
  • API-Bank（LA benchmark）：类似的跨域性能断崖

    • 🔧 方法详解

    整体框架

    CoALM 的方法分两步：构建统一数据集 CoALM-IT，然后多任务微调。

    CoALM-IT 数据集

    CoALM-IT 由三类数据组成，总计 311,583 个样本：

    数据域数据类型数据名样本数总 Token 数
    TODSingle-TurnSNIPS13,02812.3M
    LAMulti-TurnToolAce + Hammer216,319139.2M
    CRAMulti-TurnSGD (生成)82,23659.7M

    1. TOD 数据 (SNIPS)

    将 SNIPS 数据集转换为 dialogue state tracking 格式，训练模型理解用户意图并生成结构化对话状态。

    2. Function Calling 数据 (Hammer + ToolAce)

  • Hammer: 通过随机替换函数名和参数名来防止过拟合，迫使模型通过描述进行推理
  • ToolAce: 提供开放域多轮对话场景中的函数调用

    • 3. Conversational ReAct API (CRA) 数据 — 核心贡献

    CRA 是本文最重要的数据创新。基于 SGD 数据集，构造了全新的对话结构：

    $$\text{User} \rightarrow \text{Thought}_1 \rightarrow \text{Action} \rightarrow \text{Observation} \rightarrow \text{Thought}_2 \rightarrow \text{Response}$$

    关键设计：

  • Thought₁: 理解用户意图，决定是否需要调用 API
  • Action: 调用具体 API
  • Observation: API 返回结果
  • Thought₂: 分析返回结果，决定如何回应用户
  • Response: 最终用户可见的回复

    • 这种 dual-thought ReAct 结构是首次在多轮 TOD 场景中使用。使用 GPT-4o 生成 CRA 数据，并经过人工验证。

    训练细节

    基于 Llama 3.1/3.3 系列微调：

    模型基座LoRA rank训练时间GPU
    CoALM 8BLlama 3.1 8B$r=16, \alpha=32$~8h8×H100
    CoALM 70BLlama 3.3 70B$r=16, \alpha=32$~60h8×H100
    CoALM 405BLlama 3.1 405BQLoRA (nf4)-8×H100

    训练使用 LoRA，global batch size = 8，3 epochs，learning rate = $1 \times 10^{-4}$，linear warmup ratio = 0.1。

    所有三类数据混合后一次性训练，让模型在不同技能之间持续切换，避免对单一任务过拟合。

    1. 数据蒸馏问题：CRA 数据用 GPT-4o 生成，那超越 GPT-4o 的功劳是模型能力还是数据质量？

    5. 405B 仍在训练中：论文中 405B 结果标注 * 表示只完成了一个 epoch

    Agent Lightning: Train ANY AI Agents with Reinforcement Learning

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang
  • 机构: Microsoft Research
  • 时间: 2025年8月
  • 链接:
  • arXiv: 2508.03680
  • GitHub: microsoft/agent-lightning (16k+ stars)
  • 项目主页: Microsoft Research

    • 研究动机与问题

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：

    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$

  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：

    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：

    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    2. Lightning Client: 封装 Agent 运行时（agent runtime），包含两个功能模块：

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    • 关键优势：

  • Agent-agnostic 训练: Server 只关心 LLM 优化和硬件资源管理
  • Trainer-agnostic 执行: Client 端 Agent 不依赖训练框架的实现细节
  • 利用 Observability 框架: 将 OpenTelemetry 等可观测性框架引入训练流程，实现标准化的 trajectory 收集
  • 零代码修改: 支持 LangChain、OpenAI Agents SDK、AutoGen 等主流框架开发的 Agent

    核心贡献

    1. 首个完全解耦的 Agent RL 训练框架: Agent Lightning 是第一个实现 Agent 执行与 RL 训练完全分离的框架，支持任意 AI Agent 的 RL 训练，几乎零代码修改。

    2. 统一数据接口与 MDP 建模: 通过将 Agent 执行建模为 POMDP，定义了通用的数据接口，抽象了不同 Agent 的执行逻辑差异。

    3. LightningRL 层次化算法: 提出了基于 transition 分解的层次化 RL 方法，配合 Credit Assignment Module，支持多 Agent、动态工作流等复杂交互逻辑。

    4. Training-Agent Disaggregation 架构: 通过 Server/Client 分离设计，实现了训练框架与 Agent 框架的双向无关性，并引入可观测性框架提升可扩展性。

    5. 跨框架验证: 在 LangChain、OpenAI Agents SDK、AutoGen 三个主流框架上验证了通用性。

    📋 基本信息

  • 作者: Emre Can Acikgoz, Jeremiah Greer, Akul Datta, Ze Yang, William Zeng, Oussama Elachqar, Emmanouil Koukoumidis, Dilek Hakkani-Tür, Gokhan Tur

    • 🎯 研究动机与问题

    当前 LLM agent 领域存在一个核心矛盾：

    1. Task-Oriented Dialogue (TOD) 系统：擅长多轮对话管理和状态追踪，但只能处理有限的 API 集合（如 book_hotel, query_restaurant），面对新服务需要昂贵的重训练

    2. Language Agents (LA)：擅长 function calling，能处理多样化的 API，但无法维持多轮对话中的用户意图

    本文的核心问题是：能否用一个统一模型同时掌握多轮对话和工具调用？

    作者在三个 benchmark 上验证了这一矛盾的存在：

  • MultiWOZ 2.4（TOD benchmark）：LA 模型如 ToolAce 的 Success Rate 仅 18.0%
  • BFCL V3（LA benchmark）：TOD 模型如 FNCTOD 的 Overall Acc 极低
  • API-Bank（LA benchmark）：类似的跨域性能断崖

    整体框架

    CoALM 的方法分两步：构建统一数据集 CoALM-IT，然后多任务微调。

    CoALM-IT 数据集

    CoALM-IT 由三类数据组成，总计 311,583 个样本：

    数据域数据类型数据名样本数总 Token 数

    1. TOD 数据 (SNIPS)

    2. Function Calling 数据 (Hammer + ToolAce)

    • 3. Conversational ReAct API (CRA) 数据 — 核心贡献

    $$\text{User} \rightarrow \text{Thought}_1 \rightarrow \text{Action} \rightarrow \text{Observation} \rightarrow \text{Thought}_2 \rightarrow \text{Response}$$

    关键设计：

  • Thought₁: 理解用户意图，决定是否需要调用 API
  • Action: 调用具体 API
  • Observation: API 返回结果
  • Thought₂: 分析返回结果，决定如何回应用户
  • Response: 最终用户可见的回复

    • 这种 dual-thought ReAct 结构是首次在多轮 TOD 场景中使用。使用 GPT-4o 生成 CRA 数据，并经过人工验证。

    Agent Lightning: Train ANY AI Agents with Reinforcement Learning

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang
  • 机构: Microsoft Research
  • 时间: 2025年8月
  • 链接:
  • arXiv: 2508.03680
  • GitHub: microsoft/agent-lightning (16k+ stars)
  • 项目主页: Microsoft Research

    • 研究动机与问题

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：

    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$

  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：

    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：

    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    2. Lightning Client: 封装 Agent 运行时（agent runtime），包含两个功能模块：

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    • 关键优势：

  • Agent-agnostic 训练: Server 只关心 LLM 优化和硬件资源管理
  • Trainer-agnostic 执行: Client 端 Agent 不依赖训练框架的实现细节
  • 利用 Observability 框架: 将 OpenTelemetry 等可观测性框架引入训练流程，实现标准化的 trajectory 收集
  • 零代码修改: 支持 LangChain、OpenAI Agents SDK、AutoGen 等主流框架开发的 Agent
