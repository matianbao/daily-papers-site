    CoALM: A Unified Conversational Agentic Language Model

    📋 基本信息

  • 作者: Emre Can Acikgoz, Jeremiah Greer, Akul Datta, Ze Yang, William Zeng, Oussama Elachqar, Emmanouil Koukoumidis, Dilek Hakkani-Tür, Gokhan Tur
  • 机构: University of Illinois Urbana-Champaign, Oumi
  • 链接: arXiv | PDF
  • 项目主页: https://emrecanacikgoz.github.io/CoALM/

    • 🎯 研究动机与问题

    当前 LLM agent 领域存在一个核心矛盾：

    1. Task-Oriented Dialogue (TOD) 系统：擅长多轮对话管理和状态追踪，但只能处理有限的 API 集合（如 book_hotel, query_restaurant），面对新服务需要昂贵的重训练

    2. Language Agents (LA)：擅长 function calling，能处理多样化的 API，但无法维持多轮对话中的用户意图

    本文的核心问题是：能否用一个统一模型同时掌握多轮对话和工具调用？

    作者在三个 benchmark 上验证了这一矛盾的存在：

  • MultiWOZ 2.4（TOD benchmark）：LA 模型如 ToolAce 的 Success Rate 仅 18.0%
  • BFCL V3（LA benchmark）：TOD 模型如 FNCTOD 的 Overall Acc 极低
  • API-Bank（LA benchmark）：类似的跨域性能断崖

    • 🔧 方法详解

    整体框架

    CoALM 的方法分两步：构建统一数据集 CoALM-IT，然后多任务微调。

    CoALM-IT 数据集

    CoALM-IT 由三类数据组成，总计 311,583 个样本：

    数据域数据类型数据名样本数总 Token 数
    TODSingle-TurnSNIPS13,02812.3M
    LAMulti-TurnToolAce + Hammer216,319139.2M
    CRAMulti-TurnSGD (生成)82,23659.7M

    1. TOD 数据 (SNIPS)

    将 SNIPS 数据集转换为 dialogue state tracking 格式，训练模型理解用户意图并生成结构化对话状态。

    2. Function Calling 数据 (Hammer + ToolAce)

  • Hammer: 通过随机替换函数名和参数名来防止过拟合，迫使模型通过描述进行推理
  • ToolAce: 提供开放域多轮对话场景中的函数调用

    • 3. Conversational ReAct API (CRA) 数据 — 核心贡献

    CRA 是本文最重要的数据创新。基于 SGD 数据集，构造了全新的对话结构：

    $$\text{User} \rightarrow \text{Thought}_1 \rightarrow \text{Action} \rightarrow \text{Observation} \rightarrow \text{Thought}_2 \rightarrow \text{Response}$$

    关键设计：

  • Thought₁: 理解用户意图，决定是否需要调用 API
  • Action: 调用具体 API
  • Observation: API 返回结果
  • Thought₂: 分析返回结果，决定如何回应用户
  • Response: 最终用户可见的回复

    • 这种 dual-thought ReAct 结构是首次在多轮 TOD 场景中使用。使用 GPT-4o 生成 CRA 数据，并经过人工验证。

    训练细节

    基于 Llama 3.1/3.3 系列微调：

    模型基座LoRA rank训练时间GPU
    CoALM 8BLlama 3.1 8B$r=16, \alpha=32$~8h8×H100
    CoALM 70BLlama 3.3 70B$r=16, \alpha=32$~60h8×H100
    CoALM 405BLlama 3.1 405BQLoRA (nf4)-8×H100

    训练使用 LoRA，global batch size = 8，3 epochs，learning rate = $1 \times 10^{-4}$，linear warmup ratio = 0.1。

    所有三类数据混合后一次性训练，让模型在不同技能之间持续切换，避免对单一任务过拟合。

    Agent Lightning: Train ANY AI Agents with Reinforcement Learning

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang
  • 机构: Microsoft Research
  • 时间: 2025年8月
  • 链接:
  • arXiv: 2508.03680
  • GitHub: microsoft/agent-lightning (16k+ stars)
  • 项目主页: Microsoft Research

    • 研究动机与问题

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：
    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$
  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：
    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：
    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{<j}) \cdot A_j \right]$$

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    Agent Lightning: Train ANY AI Agents with Reinforcement Learning

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang
  • 机构: Microsoft Research
  • 时间: 2025年8月
  • 链接:
  • arXiv: 2508.03680
  • GitHub: microsoft/agent-lightning (16k+ stars)
  • 项目主页: Microsoft Research

    • 研究动机与问题

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：
    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$
  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：
    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：
    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{<j}) \cdot A_j \right]$$

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    UniMixer: A Unified Architecture for Scaling Laws in Recommendation Systems

    📋 基本信息

  • 作者: Mingming Ha, Guanchen Wang, Linxun Chen, Xuan Rao 等
  • 机构: 快手
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    推荐系统的 scaling law 研究目前有三大主流范式：

    1. Attention-based（如 HiFormer, FAT, HHFT）：使用 field-specific QKV 投影做异构特征交互，表达力强但计算开销大

    2. TokenMixer-based（如 RankMixer, TokenMixer-Large）：用无参数的 rule-based mixing 操作，FLOPs 友好但缺乏理论支撑

    3. FM-based（如 Wukong, Kunlun）：基于因子分解机做特征交互，可解释但受限于低阶交互

    核心问题：能否构建一个统一的 scaling 模块，融合三种范式的优势？

    🔧 方法详解

    整体框架

    UniMixer 架构由以下部分组成：Feature Tokenization → $M$ 层 UniMixer Block（含 SiameseNorm + Sparse Per-token MoE）

    Preliminaries: 三种 Scaling Block

    Heterogeneous Attention Layer

    给定输入 $X = [\boldsymbol{x}_1; \dots; \boldsymbol{x}_T] \in \mathbb{R}^{T \times D}$，异构注意力使用 token-specific 的 QKV 投影：

    $$ Q_h = \begin{bmatrix} \boldsymbol{x}_1 W^{1h}_Q \\ \vdots \\ \boldsymbol{x}_T W^{Th}_Q \end{bmatrix}, \quad K_h = \begin{bmatrix} \boldsymbol{x}_1 W^{1h}_K \\ \vdots \\ \boldsymbol{x}_T W^{Th}_K \end{bmatrix}, \quad V_h = \begin{bmatrix} \boldsymbol{x}_1 W^{1h}_V \\ \vdots \\ \boldsymbol{x}_T W^{Th}_V \end{bmatrix} $$
    $$ O_h = \text{softmax}\Big(\frac{Q_h K_h^\top}{\sqrt{d}}\Big) V_h $$

    TokenMixer

    TokenMixer 通过无参数的 split-head + concat 操作实现特征混合：

    $$ [\boldsymbol{x}_t^{(1)} | \boldsymbol{x}_t^{(2)} | \dots | \boldsymbol{x}_t^{(H)}] = \text{SplitHead}(\boldsymbol{x}_t) $$
    $$ \boldsymbol{s}^h = \text{concat}(\boldsymbol{x}_1^{(h)}, \boldsymbol{x}_2^{(h)}, \dots, \boldsymbol{x}_T^{(h)}) \in \mathbb{R}^{TD/H} $$

    约束：$H$ 必须等于 $T$（head 数 = token 数）

    Wukong (FM-based)

    $$ \text{FMB}(X) = \text{reshape}(\text{MLP}(\text{LN}(\text{flatten}(\text{FM}(X))))), \quad \text{FM}(X) = XX^\top Y $$

    其中 $Y \in \mathbb{R}^{T \times r}$ 用于降低交互矩阵 $XX^\top$ 的存储开销。

    核心创新：参数化 TokenMixer

    关键观察：TokenMixer 的 rule-based 操作可以等价表示为参数化形式。

    TokenMixer 的 mixing 操作等价于一个置换矩阵 $P$ 作用在 flatten 后的输入上：

    $$ \text{flatten}(S) = P \cdot \text{flatten}(X) $$

    其中 $P$ 是一个 双随机矩阵（doubly stochastic matrix），满足行列和均为 1。

    UniMixing 模块

    将 TokenMixer 的固定置换矩阵推广为可学习的参数化混合矩阵，同时引入 Sinkhorn-Knopp 算法约束其保持双随机性质：

    $$ \text{UniMixing}(X) = \text{reshape}(\sigma(P) \cdot \text{flatten}(X)) $$

    关键改进：

  • 移除 $H = T$ 约束：head 数不再需要等于 token 数
  • 可学习的混合模式：混合模式在训练中优化，而非预定义规则
  • 统一框架：attention、TokenMixer、FM 都是 UniMixing 在不同约束下的特例

    • UniMixing-Lite

    轻量版本，进一步压缩参数和计算：

  • 结合 attention-based 和 TokenMixer-based 的优势
  • 在保持性能的同时显著降低 FLOPs

    • SiameseNorm

    $$ \text{SiameseNorm}(\boldsymbol{x}) = \gamma \cdot \frac{\boldsymbol{x} - \mu}{\sigma + \epsilon} + \beta $$

    使用孪生归一化控制参数膨胀，其中 $\gamma$ 和 $\beta$ 在不同 head 间共享。

    📊 实验结果

    离线实验

    在快手工业级数据上验证了统一 scaling law：

    方法类型AUCUAUC参数效率
    HiFormerAttentionbaselinebaseline低
    RankMixerTokenMixercomparablecomparable中
    WukongFMcomparablecomparable中
    UniMixerUnifiedbestbest高

    Scaling Law 验证

    三种范式在统一框架下展现一致的 scaling 行为：

  • 性能随参数/FLOPs 增长呈 power law
  • UniMixer 在相同 FLOPs 下获得最佳性能
  • UniMixing-Lite 以更少参数达到接近 UniMixer 的效果

    🤔 局限性与思考

    1. 计算开销对比不清晰：RankMixer 本身几乎无额外计算，UniMixer 引入可学习矩阵后的实际开销对比缺乏详细分析

    Agent Lightning: Train ANY AI Agents with Reinforcement Learning

    基本信息

  • 作者: Xufang Luo, Yuge Zhang, Zhiyuan He, Zilong Wang, Siyun Zhao, Dongsheng Li, Luna K. Qiu, Yuqing Yang
  • 机构: Microsoft Research
  • 时间: 2025年8月
  • 链接:
  • arXiv: 2508.03680
  • GitHub: microsoft/agent-lightning (16k+ stars)
  • 项目主页: Microsoft Research

    • 研究动机与问题

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    Figure 1: Agent Lightning 框架总览。实现了 Agent 执行与 RL 训练的完全解耦。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：
    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$
  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：
    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：
    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{<j}) \cdot A_j \right]$$

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    2. Lightning Client: 封装 Agent 运行时（agent runtime），包含两个功能模块：

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    • 关键优势：

  • Agent-agnostic 训练: Server 只关心 LLM 优化和硬件资源管理

  • 利用 Observability 框架: 将 OpenTelemetry 等可观测性框架引入训练流程，实现标准化的 trajectory 收集

    1. Task-Oriented Dialogue (TOD) 系统：擅长多轮对话管理和状态追踪，但只能处理有限的 API 集合（如 book_hotel, query_restaurant），面对新服务需要昂贵的重训练

    关键设计：

  • Thought₁: 理解用户意图，决定是否需要调用 API
  • Action: 调用具体 API
  • Observation: API 返回结果
  • Thought₂: 分析返回结果，决定如何回应用户
  • Response: 最终用户可见的回复

    • 这种 dual-thought ReAct 结构是首次在多轮 TOD 场景中使用。使用 GPT-4o 生成 CRA 数据，并经过人工验证。

    GEMS: Agent-Native Multimodal Generation with Memory and Skills

    一句话总结: 提出 GEMS，首个 agent-native 多模态生成框架，将 LLM 作为认知核心（cognitive core），配备结构化记忆系统和可组合技能库，通过自主规划与执行实现跨文本/图像/音频/视频的高质量内容创作，从 model-centric 范式迈向 agent-native 范式。

    Links: arXiv | PDF | HuggingFace | Project Page

    1. 问题与动机

    核心矛盾: 生成式基础模型在各模态上已展现出色能力，但现有系统仍是 model-centric 且 narrowly specialized 的:

    1. 缺乏动态规划: 现有系统依赖固定的生成 pipeline，无法根据任务复杂度自适应调整策略

    2. 缺乏跨模态推理: 孤立的生成工具之间无法协调配合，难以完成需要多模态联合的创作任务

    3. 缺乏记忆与适应性: 不能从历史交互中学习用户偏好，每次生成都是 "从零开始"，无法积累经验

    4. 缺乏自适应工具使用: 不能根据任务需求动态选择和组合不同的生成能力

    核心 Insight: 人类创作者之所以高效，是因为能够 (1) 利用经验记忆 和 (2) 组合专业技能 来完成复杂创作。将这一范式迁移到 AI 系统 — 让 LLM 不再是单纯的生成器，而是一个会规划、委派、学习的认知核心。

    2. 方法: GEMS Framework

    GEMS 的核心设计哲学是将多模态生成从 model-centric 转为 agent-native，由三大核心组件构成:

    2.1 LLM-based Orchestrator (认知核心)

    LLM 作为中央调度者，负责:

    功能描述
    意图理解 (Intent Reasoning)解析用户的生成请求，理解真实需求
    动态规划 (Dynamic Planning)根据任务复杂度制定多步生成计划
    记忆检索 (Memory Retrieval)从记忆系统中提取相关经验和用户偏好
    技能选择 (Skill Selection)从技能库中选择合适的生成工具
    Pipeline 组合 (Pipeline Composition)动态组装生成 pipeline，串联多个技能
    质量监控 (Quality Monitoring)评估生成结果，必要时触发重试或调整

    这种设计模仿了人类专家的工作方式: 先理解需求、调用经验、选择工具、执行并验证。

    2.2 Structured Memory System (结构化记忆)

    记忆系统支持 经验积累与检索，包含:

  • 长期知识记忆 (Long-term Knowledge): 存储通用的生成知识、模态特定的最佳实践、领域知识
  • 用户偏好记忆 (User Preferences): 跟踪用户的风格偏好、常用设置、历史反馈
  • 经验记忆 (Experience Memory): 记录过往任务的成功/失败经验，包括使用了哪些技能、参数配置、生成效果

    • 记忆系统使得 GEMS 能够:

    1. 随时间积累生成经验，避免重复试错

    2. 个性化适应不同用户的创作风格

    3. 跨任务迁移知识，提高生成效率

    2.3 Modular Skill Library (模块化技能库)

    技能库是一系列 可插拔的专家工具 (pluggable expert tools)，覆盖多个模态:

    模态技能类型
    文本文案撰写、风格迁移、多语言翻译等
    图像图像生成、编辑、风格化、超分辨率等
    视频视频生成、剪辑、特效、运动合成等
    音频音乐生成、语音合成、音效创作等

    技能库的关键特性:

  • 可组合性 (Composability): 技能可以动态组合形成复杂的生成 pipeline
  • 可扩展性 (Extensibility): 新技能可以作为插件添加，无需修改核心框架
  • 动态调度: Orchestrator 根据任务需求实时选择和编排技能

    • 2.4 整体工作流

    
用户请求 → LLM Orchestrator
              ├── 意图理解 & 任务分解
              ├── 记忆检索 (相关经验 + 用户偏好)
              ├── 技能选择 & Pipeline 组合
              ├── 执行生成 (调用技能库)
              ├── 结果评估 & 可能的迭代优化
              └── 经验写回记忆系统

    3. 实验结果

    评估范围

    在跨四大模态 (文本、图像、音频、视频) 的综合 benchmark 上进行评估。

    主要发现

  • GEMS 在 质量 (quality) 和 效率 (efficiency) 两个维度上均显著超越现有 baseline
  • 持久化记忆和可组合技能对实现 robust 的 agent-native 多模态生成至关重要
  • Agent-native 范式在处理复杂、多步骤的创作任务时优势尤为明显

  • 将记忆和技能作为一等公民 (first-class components) 而非事后补丁
  • 从框架层面支持经验积累和跨任务迁移
  • 专注于创意内容生成而非通用任务执行

    • 个人评价

    优点

    1. 范式转换有洞察力: 从 model-centric 到 agent-native 的转变抓住了当前多模态生成系统的核心瓶颈 — 缺乏规划和适应能力

    2. 记忆系统设计合理: 将长期知识、用户偏好、任务经验分层存储，符合人类认知架构的直觉

    3. 技能库可组合: 模块化设计使得系统可以通过添加新技能来扩展能力，而非重新训练整个模型

    4. 全模态覆盖: 统一框架处理文本/图像/视频/音频，避免了模态孤岛问题

    5. 实用性强: 以 LLM 为核心的 orchestrator 设计与当前 AI agent 生态高度兼容

    局限 / 待讨论

    1. Orchestrator 的瓶颈: LLM 作为中央调度器，其规划能力和推理成本直接决定了系统上限和延迟

    2. 记忆系统的可扩展性: 随着经验持续积累，如何高效检索和管理大规模记忆是开放问题

    3. 技能质量参差不齐: 系统整体表现受限于最弱的技能模块（木桶效应）

    4. 评估标准: 多模态创意生成的评估本身就是难题，自动化指标可能不能完全反映创作质量

    5. 缺少具体的定量消融: 需要更详细的消融实验来量化记忆和技能各自的贡献

    关键概念

  • Agent-Native: 以 agent 为核心的系统设计范式，区别于 model-centric
  • LLM Agent: 以大语言模型为认知核心的智能体
  • Skill Library: 模块化的可组合技能/工具集合
  • Memory System: AI agent 的结构化记忆模块，支持经验积累和检索
  • Multimodal Generation: 跨模态（文本/图像/视频/音频）的内容生成
  • Orchestrator: 中央调度/编排器，负责任务规划和资源分配
  • Tool Use: LLM 调用外部工具的能力
  • Composability: 将简单组件组合成复杂系统的能力
  • 核心方法: 三层 agent 架构做多模态生成的 inference-time scaling。Agent Loop（Planner/Decomposer/Verifier/Refiner 闭环）、Agent Memory（层次化 trajectory-level 记忆）、Agent Skill（可扩展技能集合）。6B 模型套 agent 框架超越 SOTA 大模型。
  • 对比方法/Baselines: 对比: Qwen-Image, GPT；评测: GenEval2, SpatialGenEval, DPG
  • 借鉴意义: Memory 分层设计和 Skill 按需加载机制可直接借鉴。证明小模型+agent框架>大模型单推。
  • 锐评: "Agent-Native"这个词造得精准。6B 干翻 SOTA 确实漂亮。但四个 agent 每轮迭代都调 MLLM，推理成本一个字没提。Memory 压缩细节和 scalability 需要看全文确认。 🔥

    Gen-Searcher: Reinforcing Agentic Search for Image Generation

    📋 基本信息

  • 作者: Kaituo Feng, Manyuan Zhang, Shuang Chen 等
  • 机构: 未明确
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    当前图像生成模型存在一个根本限制：依赖冻结的内部知识。

    1. 生成模型的知识在训练时固定，无法处理需要最新信息的场景

    2. 知识密集型生成（如生成特定历史建筑、最新科技产品）容易出错

    3. 类似 NLP 中的 RAG 思路——生成前先搜索——在图像领域尚未被系统探索

    核心问题：能否训练一个搜索增强的图像生成 agent，通过多跳搜索收集文本知识和参考图像来指导生成？

    🔧 方法详解

    整体框架

    Gen-Searcher 是首个搜索增强的图像生成系统，将 agentic search 引入图像生成流程。

    数据构建

    构建了两个高质量数据集：

  • Gen-Searcher-SFT-10k：10k 搜索密集型 prompt + 对应 ground-truth 合成图像
  • Gen-Searcher-RL-6k：6k 样本用于 RL 训练

    • 训练流程

    两阶段训练：

    1. SFT 阶段：使用 Gen-Searcher-SFT-10k 做监督微调，让模型学会多跳推理和搜索

    2. Agentic RL 阶段：使用双 reward 反馈的 RL 训练

    双 Reward 机制

    $$ R = \alpha \cdot R_{\text{text}} + \beta \cdot R_{\text{image}} $$
  • $R_{\text{text}}$：文本 reward，评估搜索到的文本知识的相关性
  • $R_{\text{image}}$：图像 reward，评估最终生成图像的质量和准确性

    • 双 reward 结合提供更稳定和信息丰富的学习信号。

    KnowGen Benchmark

    提出 KnowGen 综合评测基准：

  • 显式要求搜索外部知识才能完成图像生成
  • 从多个维度评估模型能力

    • 📊 实验结果

    在 KnowGen benchmark 上的评测显示 Gen-Searcher 在知识密集型图像生成任务上显著超越基线方法。

    51 upvotes 表明社区对这个方向的认可。

    💡 核心贡献

    1. 首个搜索增强图像生成系统：将 RAG 思路迁移到图像生成领域

    2. 高质量数据集：Gen-Searcher-SFT-10k 和 Gen-Searcher-RL-6k

    3. 双 Reward RL：文本+图像双信号提供更稳定的训练

    4. KnowGen Benchmark：首个知识密集型图像生成评测基准

    🤔 局限性与思考

    1. 技术细节不足：摘要中方法描述过于概括，具体搜索策略的实现需要看全文

    2. 搜索延迟：实时搜索会显著增加生成延迟

    3. 搜索质量依赖：如果搜索引擎返回错误信息，可能导致生成偏差

    EvoOrch: Multi-Agent Collaboration via Evolving Orchestration

    📋 基本信息

  • 作者: Yufan Dang, Chen Qian, Xueheng Luo 等
  • 机构: Tsinghua University, SJTU, Siemens, Tencent Robotics X
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    现有 multi-agent 协作方法大多依赖静态组织结构：

    1. 预定义的拓扑（链式、星形、树形）无法随任务复杂度自适应

    2. 随着 agent 数量增长，协调开销急剧上升

    3. 手工设计拓扑费时费力且不通用

    核心问题：能否通过 RL 自动学习最优的 multi-agent 编排策略？

    🔧 方法详解

    Puppeteer 范式

    提出 "puppeteer-puppet" 范式：

  • Puppeteer（编排者）：中心化的编排 agent，动态指挥其他 agent
  • Puppets（执行者）：被编排的 agent，各有不同能力

    • Agent 三元组抽象

    每个 agent 被抽象为三元组：

    $$ \text{Agent} = (\text{model}, \text{reasoning\_pattern}, \text{tools}) $$
  • model: 底层 LLM（如 GPT-4, Claude）
  • reasoning_pattern: 推理模式（如 ReAct, CoT）
  • tools: 可用工具集合

    • RL 训练编排策略

    编排器通过 RL 持续优化，学习决定 "谁在什么时候和谁协作"：

    $$ \pi^*(\text{agent}_t | s_t) = \arg\max_\pi \mathbb{E}\left[\sum_{t=0}^T \gamma^t r_t\right] $$

    其中 $s_t$ 是当前任务状态，$r_t$ 是任务完成度 reward。

    涌现的推理结构

    分析发现，RL 优化后的编排策略涌现出更紧凑、循环式的推理结构，而非简单的链式或树形拓扑。

    📊 实验结果

    在 closed-domain 和 open-domain 场景下的评测：

    方法MMLUGSMCommonGen
    MacNetbaselinebaselinebaseline
    EvoAgentmoderatemoderatemoderate
    AutoGPTmoderatemoderatemoderate
    GPT-4 (single)goodgoodgood
    EvoOrchbestbestbest

    关键发现：编排式协作在计算成本更低的情况下实现了更优性能。

    💡 核心贡献

    1. Puppeteer 范式：首次提出中心化编排 + RL 优化的 multi-agent 协作框架

    2. 涌现式推理结构：发现 RL 优化后自动涌现紧凑循环推理模式

    3. 计算效率：以更少计算成本超越静态拓扑方法

    🤔 局限性与思考

    1. Benchmark 选择存疑：在 MMLU 这种"一个 agent 就能做"的任务上评估 multi-agent 系统，不够有说服力

    2. 中心化 vs 去中心化：Puppeteer 是单点故障，且可能成为瓶颈

    3. scalability：随 agent 数量增长，编排空间指数级膨胀

    4. 8 个 upvotes：说明实验结果可能没那么惊艳

    • 🎯 研究动机与问题

    知识蒸馏是将前沿大模型的推理能力迁移到可部署小模型的主要方法。但当前主流范式是 off-policy 的：

    1. 学生模型在静态的教师生成数据上训练

    2. 从未见过自己生成的错误 → exposure bias

    3. 训练-测试不匹配导致推理时错误累积

    🔧 方法详解

    统一 f-divergence 框架

    将所有 OPD 方法统一到 $f$-divergence 框架下：

    $$ D_f(p \| q) = \mathbb{E}_{x \sim q}\left[f\left(\frac{p(x)}{q(x)}\right)\right] $$

    不同方法对应不同的 $f$ 函数选择：

  • KL divergence ($f(t) = t\log t$): GKD
  • Reverse KL ($f(t) = -\log t$): DistiLLM
  • JSD: MiniLLM

    OPD 与 REINFORCE, PPO, GRPO 等 RL 方法的联系：

  • On-policy 采样 = RL 中的环境交互
  • 教师反馈 = reward signal

    SSD: Simple Self-Distillation for Code Generation

    📋 基本信息

  • 作者: Ruixiang Zhang, Richard He Bai, Huangjie Zheng, Navdeep Jaitly, Ronan Collobert, Yizhe Zhang
  • 机构: Apple
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    核心问题：LLM 能否仅通过自身原始输出（无 verifier、无 teacher、无 RL）提升代码生成能力？

    现有方法的复杂性：

    1. 需要 reward model / verifier 做质量筛选

    2. 需要 teacher model 做蒸馏

    3. 需要 RL 训练（如 GRPO, PPO）

    4. 这些额外组件增加了工程复杂度和计算成本

    🔧 方法详解

    SSD 方法

    极简三步流程：

    1. 采样：从模型自身采样代码解决方案（特定温度和 truncation 配置）

    2. 不做任何过滤/验证

    3. 标准 SFT 微调：直接在采样结果上做监督微调

    $$ \mathcal{L}_{\text{SSD}} = -\mathbb{E}_{x \sim p_\theta}\left[\log p_\theta(x)\right] $$

    核心发现：不筛选、不验证、直接蒸馏——反直觉地有效。

    为什么 SSD 有效？

    作者追溯到 LLM 解码中的 precision-exploration conflict：

  • Precision（精确性）：在确定性高的位置，应集中概率质量
  • Exploration（探索性）：在需要创造性的位置，应保持多样性

    • SSD 以上下文依赖的方式重塑 token 分布：

    $$ p'_\theta(x_t | x_{<t}) = \begin{cases} \text{suppress tail} & \text{precision matters} \\ \text{preserve diversity} & \text{exploration matters} \end{cases} $$
  • 在需要精确的地方，抑制 distractor tails
  • 在需要探索的地方，保留有用的多样性

    1. 极简方法：采样 + SFT，无需 verifier/teacher/RL

    2. 强效果：Qwen3-30B 在 LiveCodeBench 上 +12.9 绝对点

    3. 跨模型泛化：在 Qwen 和 Llama 的多个规模上都有效

    1. 反直觉需要更多验证："不筛选比筛了再训好"这个 claim 需要更严格的 ablation

    2. 为什么不筛选也能 work：采样本身可能隐含了一种软过滤——高温采样偏向正确模式

    4. 长期效果：多轮 SSD 是否会导致模型坍缩？

    UniMixer: A Unified Architecture for Scaling Laws in Recommendation Systems

    📋 基本信息

  • 作者: Mingming Ha, Guanchen Wang, Linxun Chen, Xuan Rao 等
  • 机构: 快手
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    推荐系统的 scaling law 研究目前有三大主流范式：

    1. Attention-based（如 HiFormer, FAT, HHFT）：使用 field-specific QKV 投影做异构特征交互，表达力强但计算开销大

    2. TokenMixer-based（如 RankMixer, TokenMixer-Large）：用无参数的 rule-based mixing 操作，FLOPs 友好但缺乏理论支撑

    3. FM-based（如 Wukong, Kunlun）：基于因子分解机做特征交互，可解释但受限于低阶交互

    核心问题：能否构建一个统一的 scaling 模块，融合三种范式的优势？

    🔧 方法详解

    整体框架

    UniMixer 架构由以下部分组成：Feature Tokenization → $M$ 层 UniMixer Block（含 SiameseNorm + Sparse Per-token MoE）

    Preliminaries: 三种 Scaling Block

    Heterogeneous Attention Layer

    给定输入 $X = [\boldsymbol{x}_1; \dots; \boldsymbol{x}_T] \in \mathbb{R}^{T \times D}$，异构注意力使用 token-specific 的 QKV 投影：

    $$ Q_h = \begin{bmatrix} \boldsymbol{x}_1 W^{1h}_Q \\ \vdots \\ \boldsymbol{x}_T W^{Th}_Q \end{bmatrix}, \quad K_h = \begin{bmatrix} \boldsymbol{x}_1 W^{1h}_K \\ \vdots \\ \boldsymbol{x}_T W^{Th}_K \end{bmatrix}, \quad V_h = \begin{bmatrix} \boldsymbol{x}_1 W^{1h}_V \\ \vdots \\ \boldsymbol{x}_T W^{Th}_V \end{bmatrix} $$
    $$ O_h = \text{softmax}\Big(\frac{Q_h K_h^\top}{\sqrt{d}}\Big) V_h $$

    TokenMixer

    TokenMixer 通过无参数的 split-head + concat 操作实现特征混合：

    $$ [\boldsymbol{x}_t^{(1)} | \boldsymbol{x}_t^{(2)} | \dots | \boldsymbol{x}_t^{(H)}] = \text{SplitHead}(\boldsymbol{x}_t) $$
    $$ \boldsymbol{s}^h = \text{concat}(\boldsymbol{x}_1^{(h)}, \boldsymbol{x}_2^{(h)}, \dots, \boldsymbol{x}_T^{(h)}) \in \mathbb{R}^{TD/H} $$

    约束：$H$ 必须等于 $T$（head 数 = token 数）

    Wukong (FM-based)

    $$ \text{FMB}(X) = \text{reshape}(\text{MLP}(\text{LN}(\text{flatten}(\text{FM}(X))))), \quad \text{FM}(X) = XX^\top Y $$

    其中 $Y \in \mathbb{R}^{T \times r}$ 用于降低交互矩阵 $XX^\top$ 的存储开销。

    核心创新：参数化 TokenMixer

    关键观察：TokenMixer 的 rule-based 操作可以等价表示为参数化形式。

    TokenMixer 的 mixing 操作等价于一个置换矩阵 $P$ 作用在 flatten 后的输入上：

    $$ \text{flatten}(S) = P \cdot \text{flatten}(X) $$

    其中 $P$ 是一个 双随机矩阵（doubly stochastic matrix），满足行列和均为 1。

    UniMixing 模块

    将 TokenMixer 的固定置换矩阵推广为可学习的参数化混合矩阵，同时引入 Sinkhorn-Knopp 算法约束其保持双随机性质：

    $$ \text{UniMixing}(X) = \text{reshape}(\sigma(P) \cdot \text{flatten}(X)) $$

    关键改进：

  • 移除 $H = T$ 约束：head 数不再需要等于 token 数
  • 可学习的混合模式：混合模式在训练中优化，而非预定义规则
  • 统一框架：attention、TokenMixer、FM 都是 UniMixing 在不同约束下的特例

    • UniMixing-Lite

    轻量版本，进一步压缩参数和计算：

  • 结合 attention-based 和 TokenMixer-based 的优势
  • 在保持性能的同时显著降低 FLOPs

    • SiameseNorm

    $$ \text{SiameseNorm}(\boldsymbol{x}) = \gamma \cdot \frac{\boldsymbol{x} - \mu}{\sigma + \epsilon} + \beta $$

    使用孪生归一化控制参数膨胀，其中 $\gamma$ 和 $\beta$ 在不同 head 间共享。

    📊 实验结果

    离线实验

    在快手工业级数据上验证了统一 scaling law：

    方法类型AUCUAUC参数效率
    HiFormerAttentionbaselinebaseline低
    RankMixerTokenMixercomparablecomparable中
    WukongFMcomparablecomparable中

    Scaling Law 验证

    三种范式在统一框架下展现一致的 scaling 行为：

  • 性能随参数/FLOPs 增长呈 power law
  • UniMixer 在相同 FLOPs 下获得最佳性能
  • UniMixing-Lite 以更少参数达到接近 UniMixer 的效果

    1. 理论统一：首次在统一数学框架下连接 attention、TokenMixer、FM 三种推荐系统 scaling 范式

    📋 基本信息

  • 作者: Zhiyuan Peng, Xuyang Wu, Huaixiao Tou, Yi Fang, Yu Gong
  • 机构: Santa Clara University
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    LLM-based 购物 agent 越来越依赖长购买历史做个性化推荐，但直接将原始历史拼接到 prompt 中存在三个问题：

    1. 噪声：购买历史中包含大量无关信息

    2. 长度：原始历史太长，超出 context window

    3. 相关性不匹配：raw history 与当前查询的相关性低

    核心问题：如何将用户购买历史蒸馏为简洁的偏好信号，提升 LLM 重排序的个性化能力？

    🔧 方法详解

    整体框架

    MemRerank 包含两个核心模块：

    1. MemAgent（Memory Extractor）：从用户购买历史中提取 query-independent 的偏好记忆

    2. LLM-based Reranker：利用偏好记忆做个性化商品重排序

    偏好记忆提取

    MemAgent 将噪声、冗长的购买记录压缩为简洁的偏好信号：

    $$ \text{Memory} = \text{MemAgent}(\text{PurchaseHistory}) $$

    关键设计：偏好记忆是 query-independent 的，不依赖当前查询，一次提取可复用于多次排序。

    1-in-5 评估任务

    构建了端到端 benchmark：给定 5 个候选商品，模型需选出用户最可能购买的 1 个。

    RL 训练

    使用 GRPO（Group Relative Policy Optimization）训练 MemAgent，以下游重排序性能作为 reward。通过 RL 优化记忆提取策略，使提取的记忆最大化下游任务表现。

    💡 核心贡献

    1. 偏好记忆框架：首次将用户购买历史蒸馏为 query-independent 偏好信号用于 LLM 重排

    2. 端到端 Benchmark：构建了 1-in-5 评估框架，同时衡量记忆质量和下游效用

    背景

    近年来 LLM 驱动的 AI Agent 在搜索、代码生成、工具调用等复杂任务中表现出色。然而，LLM 在未经专门训练的场景中（如多轮编码工作流、私有领域数据集、不熟悉的工具）仍然容易出错，无法可靠解决复杂的端到端任务。

    Reinforcement Learning 已在推理模型（如 DeepSeek-R1、Kimi k1.5）中取得了突破性进展，是优化 Agent 场景中 LLM 的强大范式。相比需要逐步标注的 Supervised Fine-Tuning，RL 仅依赖基于结果的 reward 信号，消除了对任务特定标注数据的需求。

    核心问题

    将 RL 扩展到 Agent 训练面临两大挑战：

    1. 算法层面：现有 RL 方法主要针对静态、单次调用任务（如数学推理）。Agent 的执行涉及多次 LLM 调用（不同 prompt 和 response）以及与外部工具/API 的交互，逻辑远比单轮 RL 复杂。

    2. 系统层面：现有方法将 RL 训练与 Agent 逻辑紧密耦合，或依赖序列拼接加 masking 的方式，难以支持多样化的 Agent 框架（LangChain、AutoGen、OpenAI Agents SDK 等）。

    核心思想

    Agent Lightning 的核心创新是完全解耦 Agent 执行与 RL 训练，让开发者以几乎零代码修改的方式，对任意 AI Agent 进行 RL 训练。

    方法详解

    3.1 统一数据接口 (Unified Data Interface)

    Agent Lightning 定义了一个统一的数据接口，将 Agent 执行过程中的数据标准化为 RL 训练可用的格式。

    核心概念：

  • State: Agent 执行的快照，包括程序计数器、变量值、调用栈等。形式化为：
    • $$\texttt{state}_{t}(x,k) = \left\{ \texttt{variable}_{i}^{x,k,t} \right\}_{i=1}^{V}$$
  • Semantic Variable: 被组件使用或修改的关键变量，代表关键语义（如程序意图）。不关注循环计数器等辅助变量，只关注语义层面的变化。
  • Component Invocation (Call): 语义变量的变化通过组件调用发生。第 $k$ 次执行任务 $x$ 的所有调用序列为：
    • $$\texttt{execution}(x,k) = \left\{ \texttt{call}_{i}^{x,k} \right\}_{i=1}^{N}$$

    其中每个 call 定义为：

    $$\texttt{call}_{i}^{x,k} = (\texttt{meta}_{i}^{x,k}, \texttt{input}_{i}^{x,k}, \texttt{output}_{i}^{x,k}), \quad \text{with} ~ \texttt{output}_{i}^{x,k} = C_{i}(\texttt{input}_{i}^{x,k})$$

    这里 $C_{i} \in \mathcal{M} \cup \mathcal{T}$，即组件可以是 LLM ($\mathcal{M}$) 或工具 ($\mathcal{T}$)。

  • Reward 标注: 为每个 call 分配 reward $r_i \in \mathbb{R}$，形成带 reward 的执行序列：
    • $$\texttt{execution}^{R}(x,k) = \left\{ (\texttt{call}_{i}^{x,k}, r_{i}^{x,k}) \right\}_{i=1}^{N}$$

    Figure 2: 统一数据接口示意图。左侧展示 Agent 执行流程中语义变量的更新；右侧展示对应的 trajectory 数据收集。

    3.2 Agent 中的 Markov Decision Process

    Agent Lightning 将 Agent 执行建模为 Partially Observable Markov Decision Process (POMDP)：

    $$\mathcal{M} = (\mathcal{S}, \mathcal{O}, \mathcal{A}, \mathcal{P}, \mathcal{R})$$

    各元素含义：

    符号含义
    $\mathcal{S}$状态空间，对应所有可能的 $\texttt{state}_{t}$
    $\mathcal{O}$观测空间，对应 policy LLM 的所有可能输入 $\texttt{input}_{t}$
    $\mathcal{A}$动作空间，单次 LLM 调用生成的完整 token 序列 $\texttt{output}_{t}$
    $\mathcal{T}(s's,a)$状态转移动态
    $\mathcal{R}(s,a)$奖励函数

    关键设计: 将 LLM 的一次完整生成（而非单个 token）作为一个 action，这是与传统 token-level RL 的重要区别。策略模型的目标是最大化累积回报：

    $$R = \sum_{t=1}^{T} r_t$$

    3.3 LightningRL: 层次化 RL 算法

    LightningRL 是论文提出的核心算法，解决了如何将 Agent 的多轮交互数据用于 RL 训练的问题。

    单轮 RL 回顾：现有方法（如 GRPO、REINFORCE++、PPO）主要处理单次 LLM 调用：给定任务 $x$，LLM 生成 response $\texttt{output} = (y_1, \ldots, y_N)$，获得 reward $r(x, \texttt{output}, \hat{y})$。通用训练目标为：

    $$\mathcal{L}(\theta) = -\mathbb{E}_{x \sim \mathcal{X}, \texttt{output} \sim \pi_{\theta}(\cdot|x)} \left[ \sum_{j=1}^{N} \log \pi_{\theta}(y_j | x, y_{<j}) \cdot A_j \right]$$

    其中 $A_j$ 是 token 级别的 advantage 估计。

    多轮 Agent 的挑战：

    现有方法将多轮对话拼接为单个长序列并加 mask，存在以下问题：

  • 仅适用于简单的顺序工作流
  • 无法处理多 Agent、动态工作流等复杂场景
  • 序列过长影响训练效率

    • LightningRL 的解决方案：

    LightningRL 采用层次化方法，将 trajectory 分解为独立的 transition：

    $$\texttt{execution}^{RL}(x,k) = \left\{ (\texttt{input}_{t}^{x,k}, \texttt{output}_{t}^{x,k}, r_{t}^{x,k}) \right\}_{t=1}^{T}$$

    包含一个 Credit Assignment Module，将 trajectory 级别的 return 分配到每次 LLM 调用的 reward。这使得每个 transition 可以独立用于 RL 优化，与现有的单轮 RL 算法（如 GRPO）无缝集成。

    Figure 3: LightningRL 算法示意。(a) 单轮 GRPO；(b) 现有多轮 GRPO（拼接+mask）；(c) LightningRL 将 trajectory 分解为独立 transition 进行训练。

    Automatic Intermediate Rewarding (AIR)：AIR 机制可以基于系统监控信号（如工具调用返回状态）自动为中间步骤分配 reward，缓解 Agent 训练中的 Sparse Reward 问题。

    3.4 系统设计: Training-Agent Disaggregation 架构

    Agent Lightning 的系统架构实现了训练与 Agent 执行的完全分离（TA Disaggregation）。

    Figure 4: Training-Agent Disaggregation 架构。

    两大组件：

    1. Lightning Server: 集成 RL 框架（如 VeRL），管理训练过程和 LLM 优化，对外暴露 OpenAI-like API。Server 是 Agent-agnostic 的，不关心具体的 Agent 逻辑。

    2. Lightning Client: 封装 Agent 运行时（agent runtime），包含两个功能模块：

  • 与 Server 通信，负责数据传输
  • 运行 Agent 并进行数据收集

    • 关键优势：

  • Agent-agnostic 训练: Server 只关心 LLM 优化和硬件资源管理
  • Trainer-agnostic 执行: Client 端 Agent 不依赖训练框架的实现细节
  • 利用 Observability 框架: 将 OpenTelemetry 等可观测性框架引入训练流程，实现标准化的 trajectory 收集
  • 零代码修改: 支持 LangChain、OpenAI Agents SDK、AutoGen 等主流框架开发的 Agent

    核心贡献

    1. 首个完全解耦的 Agent RL 训练框架: Agent Lightning 是第一个实现 Agent 执行与 RL 训练完全分离的框架，支持任意 AI Agent 的 RL 训练，几乎零代码修改。

    2. 统一数据接口与 MDP 建模: 通过将 Agent 执行建模为 POMDP，定义了通用的数据接口，抽象了不同 Agent 的执行逻辑差异。

    3. LightningRL 层次化算法: 提出了基于 transition 分解的层次化 RL 方法，配合 Credit Assignment Module，支持多 Agent、动态工作流等复杂交互逻辑。

    4. Training-Agent Disaggregation 架构: 通过 Server/Client 分离设计，实现了训练框架与 Agent 框架的双向无关性，并引入可观测性框架提升可扩展性。

    5. 跨框架验证: 在 LangChain、OpenAI Agents SDK、AutoGen 三个主流框架上验证了通用性。

    局限性与思考

    局限性

  • 实验规模有限: 仅在 3B 模型上验证，更大规模模型的效果尚未探索
  • 任务类型覆盖: 虽然涵盖了 text-to-SQL、RAG、math tool-use，但未涉及 GUI 交互、长期规划等更复杂的 Agent 场景
  • Credit Assignment: 论文提出了 credit assignment module，但具体的分配策略如何影响训练效果缺乏深入分析
  • 与 concatenation 方法的定量对比: Discussion 中定性分析了 transition-based 相比 concatenation-based 的优势，但缺少直接的定量 ablation

    • 个人思考

  • Agent Lightning 的解耦设计是其最大亮点。将 Agent 执行视为黑盒、仅通过标准化接口收集数据的思路，极大降低了 RL 训练的工程门槛
  • Observability 框架（OpenTelemetry）的引入很有前瞻性，将生产环境的监控基础设施复用到训练数据收集中
  • 与 RAGEN、Search-R1、Trinity-RFT 等 concatenation-based 方法的对比是后续工作的重要方向
  • 在更复杂的 Agent 场景（如 multi-agent debate、code generation pipeline）中的效果值得期待

    CoALM: A Unified Conversational Agentic Language Model

    📋 基本信息

  • 作者: Emre Can Acikgoz, Jeremiah Greer, Akul Datta, Ze Yang, William Zeng, Oussama Elachqar, Emmanouil Koukoumidis, Dilek Hakkani-Tür, Gokhan Tur
  • 机构: University of Illinois Urbana-Champaign, Oumi
  • 链接: arXiv | PDF
  • 项目主页: https://emrecanacikgoz.github.io/CoALM/

    • 🎯 研究动机与问题

    当前 LLM agent 领域存在一个核心矛盾：

    1. Task-Oriented Dialogue (TOD) 系统：擅长多轮对话管理和状态追踪，但只能处理有限的 API 集合（如 book_hotel, query_restaurant），面对新服务需要昂贵的重训练

    2. Language Agents (LA)：擅长 function calling，能处理多样化的 API，但无法维持多轮对话中的用户意图

    本文的核心问题是：能否用一个统一模型同时掌握多轮对话和工具调用？

    作者在三个 benchmark 上验证了这一矛盾的存在：

  • MultiWOZ 2.4（TOD benchmark）：LA 模型如 ToolAce 的 Success Rate 仅 18.0%
  • BFCL V3（LA benchmark）：TOD 模型如 FNCTOD 的 Overall Acc 极低
  • API-Bank（LA benchmark）：类似的跨域性能断崖

    • 🔧 方法详解

    整体框架

    CoALM 的方法分两步：构建统一数据集 CoALM-IT，然后多任务微调。

    CoALM-IT 数据集

    CoALM-IT 由三类数据组成，总计 311,583 个样本：

    数据域数据类型数据名样本数总 Token 数
    TODSingle-TurnSNIPS13,02812.3M
    LAMulti-TurnToolAce + Hammer216,319139.2M
    CRAMulti-TurnSGD (生成)82,23659.7M

    1. TOD 数据 (SNIPS)

    将 SNIPS 数据集转换为 dialogue state tracking 格式，训练模型理解用户意图并生成结构化对话状态。

    2. Function Calling 数据 (Hammer + ToolAce)

  • Hammer: 通过随机替换函数名和参数名来防止过拟合，迫使模型通过描述进行推理
  • ToolAce: 提供开放域多轮对话场景中的函数调用

    • 3. Conversational ReAct API (CRA) 数据 — 核心贡献

    CRA 是本文最重要的数据创新。基于 SGD 数据集，构造了全新的对话结构：

    $$\text{User} \rightarrow \text{Thought}_1 \rightarrow \text{Action} \rightarrow \text{Observation} \rightarrow \text{Thought}_2 \rightarrow \text{Response}$$

    关键设计：

  • Thought₁: 理解用户意图，决定是否需要调用 API
  • Action: 调用具体 API
  • Observation: API 返回结果
  • Thought₂: 分析返回结果，决定如何回应用户
  • Response: 最终用户可见的回复

    • 这种 dual-thought ReAct 结构是首次在多轮 TOD 场景中使用。使用 GPT-4o 生成 CRA 数据，并经过人工验证。

    训练细节

    基于 Llama 3.1/3.3 系列微调：

    模型基座LoRA rank训练时间GPU
    CoALM 8BLlama 3.1 8B$r=16, \alpha=32$~8h8×H100
    CoALM 70BLlama 3.3 70B$r=16, \alpha=32$~60h8×H100

    所有三类数据混合后一次性训练，让模型在不同技能之间持续切换，避免对单一任务过拟合。

    GEMS: Agent-Native Multimodal Generation with Memory and Skills

    一句话总结: 提出 GEMS，首个 agent-native 多模态生成框架，将 LLM 作为认知核心（cognitive core），配备结构化记忆系统和可组合技能库，通过自主规划与执行实现跨文本/图像/音频/视频的高质量内容创作，从 model-centric 范式迈向 agent-native 范式。

    Links: arXiv | PDF | HuggingFace | Project Page

    1. 问题与动机

    核心矛盾: 生成式基础模型在各模态上已展现出色能力，但现有系统仍是 model-centric 且 narrowly specialized 的:

    1. 缺乏动态规划: 现有系统依赖固定的生成 pipeline，无法根据任务复杂度自适应调整策略

    2. 缺乏跨模态推理: 孤立的生成工具之间无法协调配合，难以完成需要多模态联合的创作任务

    3. 缺乏记忆与适应性: 不能从历史交互中学习用户偏好，每次生成都是 "从零开始"，无法积累经验

    4. 缺乏自适应工具使用: 不能根据任务需求动态选择和组合不同的生成能力

    核心 Insight: 人类创作者之所以高效，是因为能够 (1) 利用经验记忆 和 (2) 组合专业技能 来完成复杂创作。将这一范式迁移到 AI 系统 — 让 LLM 不再是单纯的生成器，而是一个会规划、委派、学习的认知核心。

    2. 方法: GEMS Framework

    GEMS 的核心设计哲学是将多模态生成从 model-centric 转为 agent-native，由三大核心组件构成:

    2.1 LLM-based Orchestrator (认知核心)

    LLM 作为中央调度者，负责:

    功能描述
    意图理解 (Intent Reasoning)解析用户的生成请求，理解真实需求
    动态规划 (Dynamic Planning)根据任务复杂度制定多步生成计划
    记忆检索 (Memory Retrieval)从记忆系统中提取相关经验和用户偏好
    技能选择 (Skill Selection)从技能库中选择合适的生成工具
    Pipeline 组合 (Pipeline Composition)动态组装生成 pipeline，串联多个技能
    质量监控 (Quality Monitoring)评估生成结果，必要时触发重试或调整

    这种设计模仿了人类专家的工作方式: 先理解需求、调用经验、选择工具、执行并验证。

    2.2 Structured Memory System (结构化记忆)

    记忆系统支持 经验积累与检索，包含:

  • 长期知识记忆 (Long-term Knowledge): 存储通用的生成知识、模态特定的最佳实践、领域知识
  • 用户偏好记忆 (User Preferences): 跟踪用户的风格偏好、常用设置、历史反馈
  • 经验记忆 (Experience Memory): 记录过往任务的成功/失败经验，包括使用了哪些技能、参数配置、生成效果

    • 记忆系统使得 GEMS 能够:

    1. 随时间积累生成经验，避免重复试错

    2. 个性化适应不同用户的创作风格

    3. 跨任务迁移知识，提高生成效率

    2.3 Modular Skill Library (模块化技能库)

    技能库是一系列 可插拔的专家工具 (pluggable expert tools)，覆盖多个模态:

    模态技能类型
    文本文案撰写、风格迁移、多语言翻译等
    图像图像生成、编辑、风格化、超分辨率等
    视频视频生成、剪辑、特效、运动合成等
    音频音乐生成、语音合成、音效创作等

    技能库的关键特性:

  • 可组合性 (Composability): 技能可以动态组合形成复杂的生成 pipeline
  • 可扩展性 (Extensibility): 新技能可以作为插件添加，无需修改核心框架
  • 动态调度: Orchestrator 根据任务需求实时选择和编排技能

    • 2.4 整体工作流

    
用户请求 → LLM Orchestrator
              ├── 意图理解 & 任务分解
              ├── 记忆检索 (相关经验 + 用户偏好)
              ├── 技能选择 & Pipeline 组合
              ├── 执行生成 (调用技能库)
              ├── 结果评估 & 可能的迭代优化
              └── 经验写回记忆系统

    3. 实验结果

    评估范围

    在跨四大模态 (文本、图像、音频、视频) 的综合 benchmark 上进行评估。

    主要发现

  • GEMS 在 质量 (quality) 和 效率 (efficiency) 两个维度上均显著超越现有 baseline
  • 持久化记忆和可组合技能对实现 robust 的 agent-native 多模态生成至关重要
  • Agent-native 范式在处理复杂、多步骤的创作任务时优势尤为明显

  • 将记忆和技能作为一等公民 (first-class components) 而非事后补丁
  • 从框架层面支持经验积累和跨任务迁移
  • 专注于创意内容生成而非通用任务执行

    • 个人评价

    优点

    1. 范式转换有洞察力: 从 model-centric 到 agent-native 的转变抓住了当前多模态生成系统的核心瓶颈 — 缺乏规划和适应能力

    2. 记忆系统设计合理: 将长期知识、用户偏好、任务经验分层存储，符合人类认知架构的直觉

    3. 技能库可组合: 模块化设计使得系统可以通过添加新技能来扩展能力，而非重新训练整个模型

    4. 全模态覆盖: 统一框架处理文本/图像/视频/音频，避免了模态孤岛问题

    5. 实用性强: 以 LLM 为核心的 orchestrator 设计与当前 AI agent 生态高度兼容

    局限 / 待讨论

    1. Orchestrator 的瓶颈: LLM 作为中央调度器，其规划能力和推理成本直接决定了系统上限和延迟

    2. 记忆系统的可扩展性: 随着经验持续积累，如何高效检索和管理大规模记忆是开放问题

    3. 技能质量参差不齐: 系统整体表现受限于最弱的技能模块（木桶效应）

    4. 评估标准: 多模态创意生成的评估本身就是难题，自动化指标可能不能完全反映创作质量

    5. 缺少具体的定量消融: 需要更详细的消融实验来量化记忆和技能各自的贡献

    关键概念

  • Agent-Native: 以 agent 为核心的系统设计范式，区别于 model-centric
  • LLM Agent: 以大语言模型为认知核心的智能体
  • Skill Library: 模块化的可组合技能/工具集合
  • Memory System: AI agent 的结构化记忆模块，支持经验积累和检索
  • Multimodal Generation: 跨模态（文本/图像/视频/音频）的内容生成
  • Orchestrator: 中央调度/编排器，负责任务规划和资源分配
  • Tool Use: LLM 调用外部工具的能力
  • Composability: 将简单组件组合成复杂系统的能力
  • 核心方法: 三层 agent 架构做多模态生成的 inference-time scaling。Agent Loop（Planner/Decomposer/Verifier/Refiner 闭环）、Agent Memory（层次化 trajectory-level 记忆）、Agent Skill（可扩展技能集合）。6B 模型套 agent 框架超越 SOTA 大模型。
  • 对比方法/Baselines: 对比: Qwen-Image, GPT；评测: GenEval2, SpatialGenEval, DPG
  • 借鉴意义: Memory 分层设计和 Skill 按需加载机制可直接借鉴。证明小模型+agent框架>大模型单推。
  • 锐评: "Agent-Native"这个词造得精准。6B 干翻 SOTA 确实漂亮。但四个 agent 每轮迭代都调 MLLM，推理成本一个字没提。Memory 压缩细节和 scalability 需要看全文确认。 🔥

  • 作者: Kaituo Feng, Manyuan Zhang, Shuang Chen 等
  • 机构: 未明确
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    当前图像生成模型存在一个根本限制：依赖冻结的内部知识。

    1. 生成模型的知识在训练时固定，无法处理需要最新信息的场景

    2. 知识密集型生成（如生成特定历史建筑、最新科技产品）容易出错

    3. 类似 NLP 中的 RAG 思路——生成前先搜索——在图像领域尚未被系统探索

    核心问题：能否训练一个搜索增强的图像生成 agent，通过多跳搜索收集文本知识和参考图像来指导生成？

    🔧 方法详解

    整体框架

    Gen-Searcher 是首个搜索增强的图像生成系统，将 agentic search 引入图像生成流程。

    数据构建

    构建了两个高质量数据集：

  • Gen-Searcher-SFT-10k：10k 搜索密集型 prompt + 对应 ground-truth 合成图像
  • Gen-Searcher-RL-6k：6k 样本用于 RL 训练

    • 训练流程

    两阶段训练：

    1. SFT 阶段：使用 Gen-Searcher-SFT-10k 做监督微调，让模型学会多跳推理和搜索

    2. Agentic RL 阶段：使用双 reward 反馈的 RL 训练

    双 Reward 机制

    $$ R = \alpha \cdot R_{\text{text}} + \beta \cdot R_{\text{image}} $$
  • $R_{\text{text}}$：文本 reward，评估搜索到的文本知识的相关性
  • $R_{\text{image}}$：图像 reward，评估最终生成图像的质量和准确性

    • 双 reward 结合提供更稳定和信息丰富的学习信号。

    KnowGen Benchmark

    提出 KnowGen 综合评测基准：

  • 显式要求搜索外部知识才能完成图像生成
  • 从多个维度评估模型能力

    • 📊 实验结果

    在 KnowGen benchmark 上的评测显示 Gen-Searcher 在知识密集型图像生成任务上显著超越基线方法。

    51 upvotes 表明社区对这个方向的认可。

    💡 核心贡献

    1. 首个搜索增强图像生成系统：将 RAG 思路迁移到图像生成领域

    2. 高质量数据集：Gen-Searcher-SFT-10k 和 Gen-Searcher-RL-6k

    3. 双 Reward RL：文本+图像双信号提供更稳定的训练

    4. KnowGen Benchmark：首个知识密集型图像生成评测基准

    🤔 局限性与思考

    1. 技术细节不足：摘要中方法描述过于概括，具体搜索策略的实现需要看全文

    2. 搜索延迟：实时搜索会显著增加生成延迟

    3. 搜索质量依赖：如果搜索引擎返回错误信息，可能导致生成偏差

    EvoOrch: Multi-Agent Collaboration via Evolving Orchestration

    📋 基本信息

  • 作者: Yufan Dang, Chen Qian, Xueheng Luo 等
  • 机构: Tsinghua University, SJTU, Siemens, Tencent Robotics X
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    现有 multi-agent 协作方法大多依赖静态组织结构：

    1. 预定义的拓扑（链式、星形、树形）无法随任务复杂度自适应

    2. 随着 agent 数量增长，协调开销急剧上升

    3. 手工设计拓扑费时费力且不通用

    核心问题：能否通过 RL 自动学习最优的 multi-agent 编排策略？

    🔧 方法详解

    Puppeteer 范式

    提出 "puppeteer-puppet" 范式：

  • Puppeteer（编排者）：中心化的编排 agent，动态指挥其他 agent
  • Puppets（执行者）：被编排的 agent，各有不同能力

    • Agent 三元组抽象

    每个 agent 被抽象为三元组：

    $$ \text{Agent} = (\text{model}, \text{reasoning\_pattern}, \text{tools}) $$
  • model: 底层 LLM（如 GPT-4, Claude）
  • reasoning_pattern: 推理模式（如 ReAct, CoT）
  • tools: 可用工具集合

    • RL 训练编排策略

    编排器通过 RL 持续优化，学习决定 "谁在什么时候和谁协作"：

    $$ \pi^*(\text{agent}_t | s_t) = \arg\max_\pi \mathbb{E}\left[\sum_{t=0}^T \gamma^t r_t\right] $$

    其中 $s_t$ 是当前任务状态，$r_t$ 是任务完成度 reward。

    涌现的推理结构

    分析发现，RL 优化后的编排策略涌现出更紧凑、循环式的推理结构，而非简单的链式或树形拓扑。

    On-Policy Distillation Survey

    📋 基本信息

  • 作者: Mingyang Song, Mao Zheng
  • 机构: Tencent
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    知识蒸馏是将前沿大模型的推理能力迁移到可部署小模型的主要方法。但当前主流范式是 off-policy 的：

    1. 学生模型在静态的教师生成数据上训练

    2. 从未见过自己生成的错误 → exposure bias

    3. 训练-测试不匹配导致推理时错误累积

    On-Policy Distillation (OPD) 让学生模型生成自己的 trajectory 并获取教师反馈，本质上是 交互式模仿学习。

    本综述是 首个 OPD 的系统性综述。

    🔧 方法详解

    统一 f-divergence 框架

    将所有 OPD 方法统一到 $f$-divergence 框架下：

    $$ D_f(p \| q) = \mathbb{E}_{x \sim q}\left[f\left(\frac{p(x)}{q(x)}\right)\right] $$

    不同方法对应不同的 $f$ 函数选择：

  • KL divergence ($f(t) = t\log t$): GKD
  • Reverse KL ($f(t) = -\log t$): DistiLLM
  • JSD: MiniLLM

    OPD 与 REINFORCE, PPO, GRPO 等 RL 方法的联系：

  • On-policy 采样 = RL 中的环境交互
  • 教师反馈 = reward signal

    SSD: Simple Self-Distillation for Code Generation

    📋 基本信息

  • 作者: Ruixiang Zhang, Richard He Bai, Huangjie Zheng, Navdeep Jaitly, Ronan Collobert, Yizhe Zhang
  • 机构: Apple
  • 链接: arXiv | PDF

    • 🎯 研究动机与问题

    核心问题：LLM 能否仅通过自身原始输出（无 verifier、无 teacher、无 RL）提升代码生成能力？

    现有方法的复杂性：

    1. 需要 reward model / verifier 做质量筛选

    2. 需要 teacher model 做蒸馏

    3. 需要 RL 训练（如 GRPO, PPO）

    4. 这些额外组件增加了工程复杂度和计算成本

    🔧 方法详解

    SSD 方法

    极简三步流程：

    1. 采样：从模型自身采样代码解决方案（特定温度和 truncation 配置）

    2. 不做任何过滤/验证

    3. 标准 SFT 微调：直接在采样结果上做监督微调

    $$ \mathcal{L}_{\text{SSD}} = -\mathbb{E}_{x \sim p_\theta}\left[\log p_\theta(x)\right] $$

    核心发现：不筛选、不验证、直接蒸馏——反直觉地有效。

    为什么 SSD 有效？

    作者追溯到 LLM 解码中的 precision-exploration conflict：

  • Precision（精确性）：在确定性高的位置，应集中概率质量
  • Exploration（探索性）：在需要创造性的位置，应保持多样性

    • SSD 以上下文依赖的方式重塑 token 分布：

    $$ p'_\theta(x_t | x_{<t}) = \begin{cases} \text{suppress tail} & \text{precision matters} \\ \text{preserve diversity} & \text{exploration matters} \end{cases} $$
  • 在需要精确的地方，抑制 distractor tails
  • 在需要探索的地方，保留有用的多样性

    3. 是否只是 data augmentation：SSD 可能本质上是一种数据增强

    4. 长期效果：多轮 SSD 是否会导致模型坍缩？
